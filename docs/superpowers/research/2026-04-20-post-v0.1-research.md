## 1. EF Core 10 bump — blocked, wait ~6 months OR use maintainer's workaround

**Repo:** `pgvector/pgvector-dotnet`. Release cadence 2–9 months; last release
`0.3.0` on 2025-12-21.

**GitHub signal:**
- **PR #60 "Target net10.0"** (milanjaros, 2026-04-16): multi-TFM `net8.0;net10.0` +
  conditional EF 9/10 deps. **Closed unmerged** same day by maintainer `ankane`:
  > *"Thanks, but I'm not sure it's needed. Upgrading Npgsql.EntityFrameworkCore.PostgreSQL
  > in your application should fix the warning… no additional version is needed until
  > breaking changes with EF 11."*
- `roji` (Npgsql lead) pushed back, noting .NET 8/9 go EOL Nov 2026 and recommending a
  new Pgvector release aligned with EF 10. ankane acknowledged it may happen "in a few
  months, not now".
- No active work on `master` referencing EF 10 since.

**Maintainer-recommended workaround:** pin `Npgsql.EntityFrameworkCore.PostgreSQL 10.x`
directly in consuming app, suppress the NU1608 downgrade warning. Zero code change.
Risk: binary `MissingMethodException` if Npgsql.EF 10 changes an interface Pgvector.EF
calls through. (Issue #57 documented a similar past failure on EF 10-rc1.)

**Fallback if workaround breaks:** drop `Pgvector.EntityFrameworkCore` entirely, keep
base `Pgvector` package + `NpgsqlDataSourceBuilder.UseVector()`, register our own
`ValueConverter<HalfVector, byte[]>`, replace LINQ distance calls with raw
`FromSqlInterpolated` or `EF.Functions` extension wrappers on `<->`, `<=>`, `<#>`.
Call-site impact: `ChunkEntity.cs` unchanged; `HybridRetriever.cs` already uses raw
SQL (no LINQ distance calls), so no change needed — we're already using the
least-dependent API surface of Pgvector.EFCore.

**Recommendation:** **defer to v0.2** (tentative Q4 2026). Our current pin works; no
user-facing bug. When the maintainer ships `Pgvector.EFCore 0.4.0` / `10.0.0`, bump in
one commit.

## 2. AsyncFixer 2.x — take the bump now

**Repo:** `semihokur/AsyncFixer`. 2.0.0 shipped 2025-12-27, 2.1.0 on 2025-12-29.

**Breaking changes:** essentially none for consumers.
- `Microsoft.CodeAnalysis` upgraded to 4.8+ (Roslyn 4.8+); .NET 10 SDK satisfies.
- **Zero rule ID renames or removals.** `AsyncFixer01-04` keep identical IDs/semantics.
- New additive rules: `AsyncFixer05` (expanded nested-Task detection), `AsyncFixer06`
  (`Task<T>` → `Task` discard in non-async lambdas; no auto-fix).
- `AsyncFixer01` now flags async local functions + if-else terminal-await patterns →
  may surface new warnings on first build.
- `AsyncFixer02` drops several false positives (around `nameof(Task.Result)`, post-
  `Task.WhenAll` `.Result`, sync local functions/lambdas) → strictly fewer warnings.

**Current arista-mcp impact:** zero `AsyncFixer*` suppressions in `src/`. Any new
`AsyncFixer01` warnings from the expanded scope are additive; fix in place or demote
to `suggestion` via `.editorconfig`.

**Effort:** one-line bump in `Directory.Build.props:28` + `dotnet build` + `dotnet test`.
No open "breaking" / "regression" issues on the repo.

**Recommendation:** **bump now** as part of Sprint 5 v0.1.1 hygiene. Flip
`1.6.0 → 2.1.0`, handle whatever new warnings surface in-place.

## 3. FakeTimeProvider — ready to adopt, three concrete tests

**Package:** `Microsoft.Extensions.TimeProvider.Testing 10.5.0` (stable, MIT, aligns
with .NET 10 BCL). Namespace `Microsoft.Extensions.Time.Testing`.

**Key API:**
- ctor with optional `DateTimeOffset start` (default `2000-01-01T00:00:00Z`)
- `SetUtcNow(DateTimeOffset)` — absolute, throws on backward
- `Advance(TimeSpan)` — relative, fires timers + `Task.Delay` continuations
- `AdjustTime(DateTimeOffset)` — simulates NTP adjust (no timer firing)
- `AutoAdvanceAmount` — auto-tick per `GetUtcNow()`; **leave at `TimeSpan.Zero`**
  unless deliberately testing drift (silent order-dependent assertions otherwise)

**Three bugs this catches in arista-mcp (not generic testability):**

1. **Stuck ingest detection** — a future `IngestRunRepository.ListStaleRunsAsync(1h)`
   needs exact boundary behaviour. `Advance(55m)` → not stale; `Advance(10m)` → stale.
   Without FakeTimeProvider the test sleeps or is skipped, so a `>` vs `>=` bug ships.

2. **`IngestedAt` monotonicity on re-ingest** — upsert at T0, re-upsert at T0+1s. On a
   fast CI runner with real `DateTime.UtcNow`, both timestamps can collide at ms
   resolution; current "latest wins" assertion passes by luck. FakeTimeProvider makes
   the 1s gap deterministic.

3. **HybridRetriever cache expiry** (when we add caching in v0.3) — populate cache at
   T0, `Advance(ttl - 1s)` → hit; `Advance(2s)` → miss + re-query. Catches off-by-one
   on TTL boundary.

**Recommendation:** adopt in Sprint 5. Add package ref to test projects only (not
src/). Inject via base `TimeProvider` type; prod binds `TimeProvider.System` (already
done in v0.1.0), tests bind `FakeTimeProvider`.

Corrected earlier pessimism. Measured from `catalog.json`:

| Documents | 2426 |
| Total sections (all docs) | 32274 |
| Total pages across corpus | 14336 |

**Recommendation:** the bench run is a sane Sprint 5 gate — not an overnight job.
Feasible to re-run after every retrieval change.

- Smoke: 1 category at a time (`--category manual` = 225 docs, ~3 min).
- Full: all 2426 docs, ~20 min, tagged checkpoint.
- Incremental re-runs: <1 min (SHA256 early-exit skips everything).

## 5. JSON enrichment — pipeline already wired, only loader missing

**Shape of `{slug}.json`** (produced by `arista-docs/src/arista_docs/core/enrich.py`):

Top-level keys: `source_file, title, pages, num_pages, mode, chunked, char_count,
file_size, image_count, image_names, toc, sections, marker_metadata`.

- **`sections`** — authoritative `[{title, level, page_start, page_end}]`. `page_end`
  derived from the next section's `page_start - 1`.
- **`toc`** — flat `[{level, title, page?, polygon?}]`. Supersedes `sections` when you
  need hierarchy; `sections` is what we actually want.
- **`image_names`** — page-encoded (`_page_{N}_Picture_{M}.jpeg`); low-value for
  embedding; maybe a `has_image` hint per chunk eventually.
- **`marker_metadata`** — raw SDK dump; nothing extra worth consuming.

**Critical finding:** `Section.PageStart`/`PageEnd` (nullable) already exist on
`Section`, `ChunkDraft`, `ChunkEntity`, `ChunkResult`, **and the `SearchDocsTool`
output already emits `page_start`/`page_end`**. Today pages are always `null` because
nothing populates them. The entire pipeline is plumbed end-to-end; only
`DocumentLoader` is silent.

**Match strategy:** order-preserving pair of MD headings ↔ JSON sections on
`(level, normalised_title)`. Must normalise both sides with the Python contract
`enrich._clean_heading`:
- strip `**bold**`, `*italic*`, `_underscore_`, inline `<span>`/HTML
- collapse whitespace
- lowercase for comparison only (preserve original casing on output)

**Degraded modes:**
- MD heading with no JSON counterpart → null pages (no regression from today).
- JSON section with no MD match → just skipped; Nth-same-level queue handles
  duplicate titles.
- Empty `sections` array (fake fixture docs) → every chunk gets null pages; today's
  behaviour preserved.

**Effort:** ~40 LOC in `DocumentLoader`, one new internal DTO
(`EnrichedDocumentJson` / `JsonSection`), one heading-normaliser port of
`_clean_heading` (~15 LOC of regex), three new tests (stamp success, null-fallback,
MD/JSON mismatch). No DB migration.

**Recommendation:** land in Sprint 5 as the first feature after bug-fixes; users get
clickable page ranges in search results immediately.
